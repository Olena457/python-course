# Вебінар: методичні матеріали

## 1. Дидактичні принципи

1. **Спіраль, а не лекція:** кожен блок = мікроцикл «проблема → ідея → код у клітинці → видимий результат». Жодного коду без попередньо озвученої проблеми, яку він розв'язує.
2. **Одна клітинка — одна думка:** клітинка Jupyter ≤ 15–20 рядків. Якщо більше — допоміжний код виноситься в модуль `helpers/`, який показуємо оглядово ("тут нудна сантехніка, лінк у репозиторії").
3. **Результат кожні 5–7 хвилин:** аудиторія має регулярно бачити, що щось *сталося* (дані з'явились, модель заповнилась, лист прийшов).
4. **Тизер на старті:** обидві зустрічі починаються з демо кінцевого результату (30–60 сек), щоб аудиторія знала, заради чого сидить.
5. **Рівень подачі:** код зрозумілий junior+, коментарі та "чому так" — цінні для middle. Складність нарощується: HTTP-запит (знайоме) → LLM-екстракція (нове) → reasoning і агентний цикл (нове й складне).

## 2. Технічний стек (пропозиція)

| Шар | Вибір | Обґрунтування |
|---|---|---|
| Середовище | Jupyter Lab, Python 3.14, uv | наратив "клітинка → результат"; uv — швидке відтворення оточення учасниками |
| Структура | `core/` (моделі + чиста логіка, без I/O) ↔ `connectors/` (PDF, HTTP, LLM); ноутбук — диригент | чисто навіть у Jupyter; ядро тестується без мережі; «два типи конекторів» як пакети |
| Розширюваність | порти `core/ports.py` (Protocol): `VacancySource`, `MessageSender`, `MessageReceiver`, `CalendarProvider`; конектори — адаптери | заміна джерела (HN → укр. борд), каналу (email → веб-форма) чи календаря (Local → Google) = новий адаптер, без змін у ядрі |
| HTTP | httpx + beautifulsoup4 | GET до сторінки пошуку DOU та отримання лінків; HTML-текст — ідеальний матеріал |
| PDF → текст | pypdf | чистий Python, нуль системних залежностей, найпростіше для новачків |
| Валідація/структури | Pydantic v2 | центральний навчальний об'єкт: "неструктурований текст → типізована модель" (резюме і вакансії — одна техніка) |
| Стан | список об'єктів у пам'яті ноутбука | БД не потрібна: одне джерело, один запит, без дедуплікації — суттєве спрощення |
| LLM | Google Gemini через `google-genai` + `instructor` (structured output у Pydantic) | без фреймворків-агентів: мета — показати, що агент = цикл + інструменти. instructor крос-провайдерний → провайдер міняється в одному місці |
| Моделі LLM | екстракція `gemini-3.1-flash-lite`; матчинг `gemini-3.5-flash` (фолбек → Mistral); повнота даних — алгоритмічно, без LLM | різні задачі — різні моделі; LLM лише там, де судження |
| Tool use / дії | нативний tool calling; MCP — пояснюємо як стандартизацію цього ж патерну, показуємо готовий email-MCP як "продакшн-шлях" | живий цикл tool use простіше тримати в Jupyter; MCP без демо-ризику |
| Відправка відгуку | `dou_helper.py` (допоміжний скрипт) | приховуємо низькорівневий POST-запит (sessionid, csrftoken) до DOU, щоб не перевантажувати код вебінару |
| Читання відповідей | IMAP на скриньку Gmail (куди DOU надсилає сповіщення) | працює зі звичайною поштою, куди приходять відповіді від рекрутерів |
| Дані вакансій | **jobs.dou.ua** | збираємо посилання на вакансії. DOU — найбільш релевантний для України майданчик. |

## 3. Матеріали та репозиторій

```
webinar-job-agent/
├── notebooks/
│   ├── part1_agent_sees.ipynb
│   └── part2_agent_acts.ipynb
├── core/               # домен, без I/O: models.py, logic.py
├── connectors/         # краї: resume_loader.py, vacancy_source.py, llm.py
├── fixtures/           # кешовані відповіді HN/LLM — фолбек на демо
├── helpers/            # допоміжний код: dou_helper.py
├── data/
│   └── resume.pdf      # резюме "поручителя" (вхідний документ)
├── slides/             # мінімальний дек: обкладинка, схеми, CTA
├── .env                # GOOGLE_API_KEY, DOU_SESSIONID, DOU_CSRFTOKEN
└── README.md           # інструкція відтворення (uv sync → ключ → run)
```

- Репозиторій публікується учасникам після зустрічі 1 (мотивація прийти + матеріал між зустрічами).
- Слайдів мало (8–12 на зустріч): обкладинка, схема архітектури, схеми "де ми зараз", CTA. Основний екран — Jupyter.

## 6. Інтерактив (бо питань "самих по собі" мало)

- 2 опитування на зустріч у фіксованих точках (див. плани) — прості, з особистим кутом.
- Питання в чат збираються весь час, відповіді — у блоках Q&A; 2–3 "посіяні" питання тримати напоготові, якщо чат мовчить.
- Між зустрічами: міні-завдання учасникам ("проженіть екстракцію на *власному* резюме; додайте поле в `CandidateProfile`") — розбір перших 2 хв зустрічі 2.
